Ringkasan Arsitektur & Hubungan Antar Schema

Database Hera terbagi ke dalam beberapa schema PostgreSQL yang saling terintegrasi. Hal ini mengisolasi beban kerja logis, mempermudah manajemen hak akses di tingkat database (seperti Supabase/PostgreSQL), serta menyusun domain logic dengan rapi:

public
16 Tabel • 1 View

Domain inti sistem yang berisi konfigurasi menu, pembagian geografis (Region), data departemen, struktur Role/Permission (RBAC), otentikasi User, approval workflow general, serta data regional pelanggan.

moneta
9 Tabel

Domain finansial yang mengelola sinkronisasi piutang (AR Invoices) dari Accurate Online (AOL), konfigurasi limit pengingat tagihan otomatis, pinjaman sementara (PS Loans), dan settlement kas pengeluaran karyawan.

ares
4 Tabel

Domain manajemen peralatan dan armada mesin (Machine solutions), melacak manufaktur, model, serial number mesin, status jam operasi (Hour Meter), serta lokasi kerja (site) masing-masing unit.

application_logs
4 Tabel

Domain audit log yang didedikasikan untuk pencatatan riwayat aktivitas pengguna (User Activity), sinkronisasi scheduler system ke AOL, logs pengiriman reminder invoice, serta notifikasi terpusat (Helios).

Relasi dan Ketergantungan Schema

Sistem ini dirancang dengan prinsip modularitas tinggi namun terhubung erat pada entitas User dan Customer:

  • Otentikasi & Keamanan (RBAC): Hubungan multi-to-multi antara t_core_users, t_cfg_roles, dan t_cfg_permissions dikontrol melalui tabel relasional.
  • Alur Kerja Persetujuan (General Approvals): Modul t_core_approvals dihubungkan secara dinamis ke entitas modul lain (misal moneta.t_ps_loans atau moneta.t_ps_settlements) lewat field Module dan EntityID yang berfungsi sebagai Polymorphic Relation.
  • Integrasi Pihak Ketiga (Accurate Online): Sinkronisasi data diatur secara berkala dengan memetakan ID unik AOL (seperti aol_id, customer_aol_id) di tabel Master Data t_md_customers dan moneta.t_ar_invoices.

VIEW

public.v_md_customers (Database View)

View khusus database yang menggabungkan master data customer dengan kalkulasi total outstanding piutang yang dikumpulkan dinamis dari invoice yang belum lunas (UNPAID/PARTIAL) di modul moneta.

💡

Informasi Teknis: View ini mempermudah proses query di frontend untuk menampilkan dasboard piutang tanpa perlu melakukan JOIN/SUM berat berulang kali di tingkat REST API.

Query SQL Definisi View:

SELECT 
    c.id, c.created_at, c.updated_at, c.deleted_at,
    c.aol_id, c.customer_no, c.name, c.email, c.phone, c.address,
    c.npwp, c.category, c.province_id, c.city_id, c.tax_office,
    c.pkp, c.credit_limit, c.sync_at,
    COALESCE(inv.total_outstanding, 0) as total_outstanding,
    COALESCE(inv.total_overdue, 0) as total_overdue,
    COALESCE(inv.open_invoice_count, 0) as open_invoice_count
FROM 
    t_md_customers c
LEFT JOIN (
    SELECT 
        customer_aol_id,
        SUM(balance_amount) as total_outstanding,
        SUM(CASE WHEN due_date < CURRENT_DATE THEN balance_amount ELSE 0 END) as total_overdue,
        COUNT(*) as open_invoice_count
    FROM 
        moneta.t_ar_invoices
    WHERE 
        status != 'PAID'
    GROUP BY 
        customer_aol_id
) inv ON c.aol_id = inv.customer_aol_id;
