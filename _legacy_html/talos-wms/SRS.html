01

Pendahuluan

1.1 Tujuan Dokumen

Dokumen Software Requirements Specification (SRS) ini bertujuan untuk menjabarkan spesifikasi teknis, kebutuhan fungsional, perilaku sistem (behavior), aturan aplikasi (business rules), serta validasi yang harus diimplementasikan pada TALOS Warehouse Management System (WMS). Dokumen ini menjadi acuan utama bagi tim pengembang perangkat lunak, QA, dan pemangku kepentingan berdasarkan referensi dari Product Requirements Document (PRD).

1.2 Ruang Lingkup Sistem

TALOS WMS adalah sistem manajemen gudang terpadu yang memiliki kapabilitas:

  • Manajemen Inbound, Outbound, Transfer, dan Stock Adjustment.
  • Penggunaan Barcode Serialized untuk penelusuran histori setiap unit barang.
  • Visualisasi Layout Gudang 2D yang interaktif.
  • Mendukung penyimpanan Multi-Level (palet, dus, satuan) dengan struktur rack dan bin management.
  • Proses Assembly dan Disassembly berdasarkan Bill of Materials (BOM).
  • Stock Opname (cycle count & full count) dengan rekonsiliasi otomatis.
  • Terhubung secara sinkron (dua arah) dengan Accurate Online.
  • Pelaporan operasional dan analitik performa gudang.

Deskripsi Umum

2.1 Karakteristik Pengguna (Persona)

🏭
Orang Gudang
Melakukan transaksi fisik, membutuhkan UI yang responsif dan mobile-friendly (tombol besar, scanning mudah, input efisien).
🗂️
Admin Spare Part
Melakukan manajemen master data, konfigurasi BOM, manajemen layout gudang, pengaturan master lokasi, serta penanganan stock opname.
📊
Parts Analyst
Membutuhkan laporan lengkap, analitik historis, performa KPI, dan dashboard operasional.
🔐
Super Admin / IT
Memegang kontrol akses tertinggi, konfigurasi integrasi Accurate Online, memonitor log audit, dan resolusi konflik sinkronisasi.

2.2 Lingkungan Operasi & Tech Stack

💻
Frontend

Web browser modern di desktop, tablet, dan smartphone. Dibangun menggunakan Next.js (React), TypeScript, TailwindCSS, Zustand, Konva.js (Canvas 2D), dan react-zxing (Barcode Scanner).

⚙️
Backend

Layanan API berbasis Go (Golang) v1.22+ menggunakan framework Gin/Echo, dengan basis data PostgreSQL. Message Queue menggunakan RabbitMQ dan Redis untuk cache. Autentikasi via JWT.

🔄
Integrasi

API Accurate Online v1 (REST) via autentikasi OAuth2 dan Webhook Endpoint.

03

Kebutuhan Fungsional & Aturan Aplikasi

Bagian ini menjelaskan secara detail spesifikasi interaksi sistem dengan penekanan pada logika, perilaku, dan aturan validasi data.

3.1 Manajemen Barcode Serialized

Behavior & Aturan

Keunikan SN : Setiap unit satuan terkecil harus memiliki Serial Number (SN) yang bersifat unik secara global di dalam sistem.
Siklus Hidup : Sistem mengawasi siklus hidup SN: dari penerimaan (Available) → mutasi (In-Transit / Available) → pengeluaran/assembly (Consumed/Disposed).
Format SN : [KodeGudang]-[KodeBarang]-[YYYYMMDD]-[SequenceUrut].
Scan & Mapping : Jika perusahaan menerima barcode pabrik, fitur Scan & Mapping dapat mendaftarkan SN bawaan pabrik selama nilainya unik.
🛡️
Validasi:
- Pembuatan/Penerimaan ganda: Jika di-scan SN yang sudah ada dengan status masih ada di gudang, munculkan error blokir: "Serial Number [SN] sudah tercatat aktif di sistem."
- Scan pengeluaran: Sistem memvalidasi apakah SN sedang diblokir (Hold) atau di-reserve (sudah masuk ke keranjang pesanan/WO lain).

3.2 Racking, Bin, & Multilevel Storage

Behavior & Aturan

Manipulasi Lokasi : Struktur lokasi tidak boleh dihapus atau diubah dimensinya secara drastis jika masih berisi barang aktif.
Kapasitas Bin : Setiap bin memiliki aturan Kapasitas Maksimal (berdasarkan kuantitas unit, volume, atau berat).
Putaway Suggestion : Sistem menggunakan skoring aturan: 1) Bin yang sudah berisi jenis barang yang sama (FIFO), 2) Zone yang sesuai dengan kategori barang, 3) Kapasitas mencukupi.
Level Scanning : Bersifat hirarkis (contoh: memindai SN Palet akan memutasi seluruh SN Kardus dan SN Satuan di dalamnya secara bersamaan).
🛡️
Validasi:
- Jika Putaway melebihi batas kapasitas Bin, tampilkan Soft Warning untuk administrator/supervisor: "Kapasitas Bin [NamaBin] melebihi batas. Override persetujuan dibutuhkan."
- Bin berstatus Blocked/Karantina menolak seluruh transaksi masuk (Inbound) dan keluar (Outbound).

3.3 Transaksi Gudang Dasar

Behavior & Aturan

Goods Receipt (Inbound) : Mendukung penerimaan berbasis Purchase Order dari Accurate atau penerimaan langsung (Direct Receipt). Sistem akan merilis daftar SN baru dan mencetak label (bila diperlukan).
Goods Issue (Outbound) : Mendukung pengeluaran berbasis Sales Order atau permintaan inter-departemen. Disarankan menggunakan metode FIFO secara systematic alert.
Transfer Gudang : Wajib menerapkan mekanisme "Two-Step Transfer". Gudang asal memproses pengeluaran, gudang tujuan memproses penerimaan (In-Transit state).
Adjustment : Perubahan stok paksa. Semua adjustment wajib mengisi alasan (Damage, Lost, Found, Expired).
🛡️
Validasi:
- Penerimaan barang di luar cakupan referensi (PO) tidak valid (jika referensi ditetapkan mutlak).
- Pengeluaran divalidasi hingga level SN: "Item tidak ditemukan pada lokasi terkait" atau "Stok defisit".
- Transfer tidak valid jika gudang tujuan diatur dalam status Inaktif.

3.4 Assembly & Disassembly (BOM)

Behavior & Aturan

Assembly : Menggabungkan berbagai part ke produk jadi berdasarkan nested BOM. Sistem mendepresiasi stok komponen dan meng-generate SN baru untuk barang jadi.
Disassembly : Pemisahan produk jadi kembali menjadi serpihan part asal berdasar BOM. Komponen dapat di-set status kondisinya (Cacat/Bagus) usai disassembly.
Work Order (WO) : Prosedur dikelola melalui dokumen Work Order dengan 3 state: Pending, In-Progress, Completed.
🛡️
Validasi:
- Tidak dapat merubah status WO menjadi In-Progress bila ketersediaan kuantitas bahan baku komponen kurang dari persyaratan BOM.
- Bahan baku yang dimasukkan pada WO In-Progress otomatis berstatus Reserved agar tidak terpakai oleh transaksi lain.

3.5 Stock Opname (Rekonsiliasi Fisik)

Behavior & Aturan

Mode Pelaksanaan : Mendukung dua mode pelaksanaan: Full Count (mengunci operasional gudang total) dan Cycle Count (hanya lokasi/zona tertentu).
Multi-Petugas : Memungkinkan pendelegasian area ke beberapa user scanner berbeda (multi-petugas).
Rekonsiliasi Otomatis : Sistem membandingkan data hitung manual dengan data on-hand. Jika muncul variance, data dilempar ke approval flow supervisor.
🛡️
Validasi:
- Lokasi (Bin) tidak dapat memulai siklus Stock Opname jika masih ada tugas Picking atau transaksi gantung yang belum diselesaikan.
- Tidak dapat memposting (menyetujui) Stock Opname sebelum semua variance telah diselidiki dan diizinkan oleh Supervisor / Super Admin.

3.6 Layout Gudang 2D (Visualisasi UI)

Behavior & Aturan

Rendering Heatmap : Dirender secara real-time berbasis perhitungan kapasitas persentase masing-masing Bin.
Inspektor Detail : Antarmuka mengizinkan klik elemen Bin untuk meluncurkan inspektor detail stok.
Drag-and-Drop : Pemindahan posisi Bin hanya diperbolehkan bila pengguna memiliki role Super Admin / Admin Spare Part (Edit Mode aktif).
04

Antarmuka Eksternal (Integrasi Accurate Online)

Aturan & Behavior Sinkronisasi

Protokol Standar : Sinkronisasi dua arah dilakukan menggunakan standar OAuth2 dan REST API v1 milik Accurate Online.
Trigger Real-time : Data yang mempengaruhi nilai aset atau finansial (GRN, DO, Adjustment, Assembly conversion) WAJIB terdorong secara real-time (atau diantrikan bila offline).
Sync Master Data : Master Barang tersinkronisasi otomatis dari Accurate Online ke WMS melalui scheduler (polling) dan Webhook on-demand.
Message Queue : Seluruh antrian transaksi dikelola oleh perantara Message Queue (RabbitMQ).

Validasi Integrasi (Conflict Resolution)

Rate Limiting : Menangani API Rate Limiting dari Accurate Online dengan melakukan pengereman request (Throttling / Exponential Backoff).
Dead Letter Queue : Transaksi gagal (karena kendala jaringan atau data tidak lengkap) dipindahkan ke Dead Letter Queue (DLQ), dimunculkan sebagai tanda merah (Sync Failed) di dashboard admin WMS.
Manual Retry : Admin wajib melakukan intervensi perbaikan data sebelum melakukan Manual Retry.
