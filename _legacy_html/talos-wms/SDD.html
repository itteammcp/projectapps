01

Pendahuluan

1.1 Tujuan Dokumen

Software Design Document (SDD) ini memberikan gambaran komprehensif terkait desain sistem pada TALOS WMS. Dokumen ini menjadi pedoman implementasi bagi developer (Backend dan Frontend), DevOps, serta Database Administrator untuk memastikan bahwa perangkat lunak yang dibangun selaras dengan spesifikasi fungsional dan teknis yang tercantum pada PRD dan SRS.

1.2 Komponen Desain Utama

Komponen Deskripsi
Arsitektur Gambaran level tinggi dari infrastruktur, integrasi, dan komunikasi data.
Database Desain skema data, struktur relasi, dan strategi penyimpanan.
API Standar komunikasi antar layanan (REST/RPC).
Struktur Backend Pola desain, layer aplikasi, dan manajemen queue.
Struktur Frontend Pengelolaan state, routing, dan integrasi UI Canvas 2D.
02

Arsitektur Sistem

2.1 Arsitektur High-Level

TALOS WMS menggunakan arsitektur Client-Server (Monolith Modular/Microservices-ready) berbasis cloud. Lapisan presentasi dan lapisan logika bisnis dipisahkan melalui antarmuka REST API dan WebSocket.

🖥️
Client Layer

Aplikasi web responsif berbasis Next.js (React) dan TypeScript. Berfungsi sebagai antarmuka untuk Orang Gudang (Mobile) hingga Super Admin (Desktop).

⚙️
API Gateway / Application Layer

Service utama dibangun dengan Golang. Menangani routing, load balancing dasar, autentikasi (JWT), dan bisnis logik WMS.

🗄️
Data Storage Layer

PostgreSQL sebagai relational database utama (RDBMS). Redis digunakan untuk caching, session management, dan real-time states.

📨
Message Broker / Integration

RabbitMQ digunakan untuk mengelola antrian asinkron, terutama untuk komunikasi data dengan sistem eksternal (Accurate Online API).

2.2 Alur Integrasi (Accurate Online)

Integrasi dengan Accurate Online dilakukan melalui mekanisme Sinkronisasi Dua Arah:

Arah Sinkronisasi Mekanisme & Proses
Inbound (Accurate ke WMS) Webhook atau Polling (Cron job pada Golang) menarik Master Barang, Purchase Order, dan Sales Order. Data disimpan dalam database internal untuk proses WMS.
Outbound (WMS ke Accurate) Transaksi (GRN, DO, Adjustment, Assembly) di-push ke Message Queue (RabbitMQ). Worker background mengambil pesan dan memanggil API Accurate Online untuk membukukan transaksi secara finansial.
Handling Error Mekanisme Dead Letter Queue (DLQ) digunakan apabila request gagal, yang kemudian memunculkan notifikasi kepada Super Admin untuk di-retry.
03

Desain Database

Database utama menggunakan PostgreSQL dengan nama database hera. Semua tabel operasional disimpan di dalam dua schema utama: schema talos (talos.*) untuk fungsionalitas WMS dan schema logix (logix.*) khusus untuk fungsi Logistik (Logi-X). Tabel global seperti users dan roles berada di schema public.

3.1 Kategori & Konvensi Penamaan Tabel

Seluruh tabel dalam schema talos menggunakan awalan t_ yang diikuti dengan kode kategori modul.

Kategori (Prefix) Keterangan Contoh Tabel
t_core_ Data master operasional & sistem inti WMS. t_core_warehouses, t_core_items
t_cfg_ Konfigurasi dan pengaturan (termasuk permission jika ada spesifik). t_cfg_permissions
t_ibd_ Transaksi Inbound (Penerimaan Barang). t_ibd_receipts
t_obd_ Transaksi Outbound (Pengeluaran Barang). t_obd_deliveries
t_inv_ Manajemen Inventory, Stok, dan Serial Number. t_inv_serial_numbers

3.2 Skema Detail Tabel

1. Tabel talos.t_core_warehouses

Field Tipe Data Constraint Deskripsi
id UUID PK ID Unik Gudang
code VARCHAR(20) UNIQUE, NOT NULL Kode gudang (cth: WH01)
name VARCHAR(100) NOT NULL Nama gudang
address TEXT - Alamat fisik gudang
status VARCHAR(20) DEFAULT 'ACTIVE' Status operasional gudang

2. Tabel talos.t_core_bins (Lokasi Penyimpanan)

Field Tipe Data Constraint Deskripsi
id UUID PK ID Unik Bin
rack_id UUID FK (t_core_racks.id) Referensi rak tempat bin berada
code VARCHAR(50) UNIQUE, NOT NULL Kode lokasi (cth: WH01-A-R1-L1-B1)
max_capacity INT DEFAULT 0 Kapasitas maksimal (unit)
status VARCHAR(20) DEFAULT 'AVAILABLE' AVAILABLE, BLOCKED, FULL

3. Tabel talos.t_core_items (Master Barang)

Field Tipe Data Constraint Deskripsi
id UUID PK ID Unik Barang
accurate_id BIGINT UNIQUE ID barang dari Accurate Online
code VARCHAR(50) UNIQUE, NOT NULL SKU / Kode barang
name VARCHAR(200) NOT NULL Nama lengkap barang
uom VARCHAR(20) NOT NULL Unit of Measurement (PCS, BOX)

4. Tabel talos.t_inv_serial_numbers

Field Tipe Data Constraint Deskripsi
id UUID PK ID Internal SN
sn_code VARCHAR(100) UNIQUE, NOT NULL Barcode Serial Number
item_id UUID FK (t_core_items.id) Barang terkait
bin_id UUID FK (t_core_bins.id) Lokasi bin saat ini
status VARCHAR(20) NOT NULL AVAILABLE, RESERVED, CONSUMED, IN-TRANSIT

5. Tabel talos.t_ibd_receipts (Inbound)

Field Tipe Data Constraint Deskripsi
id UUID PK ID Transaksi Penerimaan
doc_number VARCHAR(50) UNIQUE, NOT NULL Nomor dokumen GRN
warehouse_id UUID FK (t_core_warehouses.id) Gudang penerima
received_date TIMESTAMP NOT NULL Waktu barang diterima
sync_status VARCHAR(20) DEFAULT 'PENDING' PENDING, SYNCED, FAILED

6. Tabel talos.t_obd_deliveries (Outbound)

Field Tipe Data Constraint Deskripsi
id UUID PK ID Transaksi Pengeluaran
doc_number VARCHAR(50) UNIQUE, NOT NULL Nomor dokumen DO
warehouse_id UUID FK (t_core_warehouses.id) Gudang pengeluar
shipped_date TIMESTAMP NOT NULL Waktu barang dikeluarkan
sync_status VARCHAR(20) DEFAULT 'PENDING' PENDING, SYNCED, FAILED

3.3 Skema Database Modul Logi-X (Schema: logix)

Tabel-tabel berikut merupakan bagian dari modul pengiriman (Logistics Execution). Catatan Penting: Berbeda dengan WMS yang menggunakan schema talos, seluruh tabel Logi-X dipisahkan ke dalam schema mandiri yaitu schema logix (logix.*) di dalam database yang sama.

1. Tabel Master Data Logistik

Nama Tabel Keterangan Fungsi
t_md_vendors Master data mitra vendor/kurir, menyimpan SLA scan DO dan referensi ke ID AOL.
t_md_rate_destinations Data area / kota yang berfungsi sebagai titik origin dan destination.
t_md_vendor_rates Tarif kurir berdasarkan origin, destination, transportasi, dan SLA pengiriman.
t_md_do_return_slas SLA pengembalian fisik DO berdasarkan rute tujuan (origin to destination).
t_md_aol_transactions Menyimpan riwayat dan data sinkronisasi transaksi SO/PO dari Accurate Online.

2. Tabel Transaksi Pengiriman (Shipments)

Nama Tabel Keterangan Fungsi
t_dom_shipments Header transaksi pengiriman (AWB, vendor, status, tanggal aktual, SLA metrics).
t_dom_shipment_details Detail kustomer, berat aktual & volumetrik, referensi AOL, dan file DO scan.
t_dom_shipment_trackings Riwayat pergerakan logistik (event tracking API atau input manual).
t_dom_shipment_invoices Data tagihan invoice jasa logistik, beserta komparasi biaya dan efisiensi.
💡
Strategi Indexing: Indexing pada kolom sn_code, code (pada items), dan bin_id bersifat esensial untuk menjamin waktu query < 2 detik pada saat scanning massal. Indexing pada tabel Logi-X difokuskan pada awb_no, shipment_number, dan transaction_number.
04

Desain API

TALOS WMS mengekspos RESTful API (dan WebSocket untuk operasi real-time). Seluruh API wajib menggunakan Header Authorization Bearer [Token].

4.1 Konvensi Endpoint RESTful

Konvensi Standar / Keterangan
Base URL /api/v1/
Format Repons Terstandarisasi { "status": 200, "message": "Success", "data": {...} }
Error Handling Error terstruktur dengan kode bisnis yang spesifik (misal: ERR_SN_DUPLICATE, ERR_BIN_FULL).

4.2 Modul Endpoint Utama

Modul Method & Endpoint (Contoh) Fungsi
Auth POST /api/v1/auth/login Autentikasi dan penerbitan JWT
Master Data GET /api/v1/items
POST /api/v1/locations/bins		 Manajemen data barang, gudang, dan lokasi
Transaksi POST /api/v1/transactions/inbound
POST /api/v1/transactions/transfer 		Memproses penerimaan dan pemindahan barang dengan payload SN
Inventory GET /api/v1/inventory/bins/:id
GET /api/v1/inventory/serial/:sn		 Pencarian stok, kapasitas Bin, dan penelusuran riwayat SN
Assembly POST /api/v1/assembly/work-orders Membuat WO berdasarkan BOM, reserve komponen, merilis SN baru
Sync POST /api/v1/sync/retry-failed Manual trigger retry transaksi gagal dari DLQ
Logi-X Master Data GET /api/v1/logix/vendor-rates
POST /api/v1/logix/destinations		 Mengelola tarif kurir, SLA DO, dan destinasi rute.
Logi-X Shipments POST /api/v1/logix/shipments
GET /api/v1/logix/shipments/:id/tracking 		Operasional inbound/outbound, melacak status armada, dan upload DO.
Logi-X Invoices GET /api/v1/logix/invoices
POST /api/v1/logix/invoices/generate		 Kalkulasi tagihan invoice kurir, perbandingan efisiensi tarif pengiriman.
05

Struktur Backend

5.1 Pola Clean Architecture (Golang)

Pengembangan backend menggunakan bahasa Go (Golang) dan mengikuti pola Clean Architecture (Domain-Driven Design). Struktur kode dibagi ke dalam layer yang terisolasi:

cmd/ : Entry point aplikasi (main.go), inisialisasi server, dan worker.
internal/delivery/ : Lapisan presentasi API. Mengandung Router (Gin/Echo), Handler/Controller HTTP, dan Middleware (Auth, Logger, CORS).
internal/usecase/ : Logic bisnis aplikasi. Mengorkestrasi aliran data, validasi bisnis (seperti pengecekan kapasitas Bin, verifikasi status SN), dan manajemen transaksi (DB transaction).
internal/repository/ : Interaksi langsung dengan database PostgreSQL (menggunakan SQL/ORM seperti GORM/sqlx) dan redis.
internal/domain/ : Mendefinisikan tipe entitas utama (struct) dan antarmuka (interface) yang dipakai di Repository dan Usecase.
pkg/ : Library utilitas eksternal yang dapat digunakan ulang (contoh: Konfigurasi Accurate API Client, utilitas generator Barcode, enkripsi).
⚡
Kinerja & Konkurensi:
Proses ekspor laporan PDF/Excel dan antrian Push API Accurate dijalankan secara Asynchronous menggunakan Goroutines dan Message Broker (RabbitMQ) agar tidak memblokir response thread utama.
06

Struktur Frontend

6.1 Next.js App Router Architecture

Aplikasi klien menggunakan Next.js (App Router) dengan bahasa TypeScript. Struktur difokuskan pada manajemen rute hierarkis dan pengalaman pengguna tingkat tinggi.

🗂️
Struktur Folder

Menggunakan pendekatan fitur-driven (app/dashboard, app/inventory, app/assembly). Komponen modular disimpan di folder components/.

🧠
State Management

State global untuk sesi user, notifikasi, dan active-tab dikelola oleh Zustand. Caching request API menggunakan React Query atau SWR.

📱
Interaksi Hardware

Modul scanner menggunakan integrasi react-zxing untuk membaca via kamera perangkat dan focus-trap untuk scanner Bluetooth (HID Mode).

🗺️
Canvas 2D Rendering

Visualisasi gudang (Layout 2D) menggunakan library Konva.js atau React Konva, yang memungkinkan Drag-and-Drop elemen Rak serta rendering data heatmap stok (Canvas API).

6.2 Penanganan Offline & UI/UX

Aspek Pendekatan / Implementasi
Optimistic UI Transaksi di frontend diasumsikan sukses untuk memberi respon visual yang cepat (di bawah 2 detik), sembari server memvalidasi di background.
Dark Mode & Tema Didukung menggunakan TailwindCSS (class dark), desain standar Tokens diatur dalam globals.css sesuai dengan styling panduan perusahaan.
Aksesibilitas Mobile Antarmuka Orang Gudang menggunakan tata letak layar penuh yang teroptimasi, tombol berukuran minimum 44px, dan peringatan visual yang kontras tinggi.
